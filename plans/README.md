## Findings considered and rejected (do not re-audit)

- **"Whisper temp-file leak on transcription failure"** — false positive. The
  `defer` cleanup in `MeetingStore.swift` `stopMeetingRecordingAndSave()` is
  installed at the top of the function (before the Whisper loop) and runs on every
  exit path; Whisper calls also have inner try/catch.
- **Most "N+1" perf claims in calendar/base.py** — the queries already use
  `.select_related("project", "state")`; the calendar-tasks query is bounded by the
  client's date range, not unbounded. Not worth a plan (a minor note: the date range
  is client-controlled, so an extreme range is possible — low value).
- **"Divergent HTML escaping = security risk"** — false premise. Both `chat.py`
  (`html.escape`) and `calendar/base.py` (`django.utils.html.escape`) escape
  correctly. The duplication is real but maintainability-only — captured as plan 003
  at P3, reframed honestly.
- **"OAuth token in logs"** — the current code does not log tokens; defensive-only,
  not evidenced. Not planned.
- **Title-generation race / memory-update transaction / missing deinit on Tasks** —
  real but low-impact (title is renameable; memory writes are upserts; Task callbacks
  use `[weak self]`). Recorded; not worth dedicated plans now.
- **MCP auth-header encryption, agent-chat permission decorators, live-server Zod
  validation, chrome-extension token storage, document HTML sanitization** — audited
  and confirmed correct (no issue).

## Real product bugs surfaced by 007 (candidates for plan 008)

These are the residual failures after 007 — genuine product bugs (or stale tests),
NOT papered over. Evidence is the failing test that exposes each.

1. **Cycle create ignores URL `project_id`** (`test_cycles.py`, 3 fails) — `CycleCreateSerializer.validate()`
   reads `project_id` from `initial_data`, but the view (`plane/api/views/cycle.py:308`)
   passes only `context={"request": request}` and never injects `project_id` → 400
   "Project ID is required". Real bug.
2. **API-token PATCH can modify service tokens** (`test_api_token.py`) — the `patch()`
   queryset doesn't filter `is_service=False` (the `delete()` path does). Owner can
   patch a service token. Real bug.
3. **AI-connectors `secret` field crash** (`test_ai_connectors_app.py`, 2) —
   `WorkspaceAIConnectorSerializer` has a writable `secret` with no model field
   (model uses `secret_encrypted`); `create(**validated_data)` → `TypeError`. Real bug.
4. **Google-Drive attachment 409 id type mismatch** (`test_google_drive_attachment_app.py`) —
   409 response returns `id` as str, 201 returns a UUID; equality assert fails. Serializer
   field-type inconsistency. Likely real.
5. **copy_s3 task NOT NULL on `description_json`** (`test_copy_s3_objects.py`) — bg task
   `issue.save()` hits a NOT NULL constraint; also exercises real S3. Real bug + needs mock.
6. **Stale tests needing a ProjectMember / 403-vs-404 order** (`test_issue_app.py` x2,
   `test_page_app.py` x2, `test_project_bookmark_app.py`) — tests create a `Project`
   without a `ProjectMember`, so `ProjectEntityPermission` returns 403 before the view's
   404 lookup. Either test setup gaps (add ProjectMember) or an intended permission-order
   decision. Lower-stakes; verify intent before changing.

Recommend plan 008: fix items 1–5 (real bugs, each with the failing test as its
regression gate), and resolve item 6's tests. After 008, plan 005's "full suite green"
gate becomes achievable.

**008 DONE (merged `8a5ece2359`)** — all 5 real bugs fixed (each gated by its test),
plus 3 item-6 stale-test fixes. Suite 13→2. Two follow-ups surfaced:

- **009 — `allow_permission` 403-before-404 (cross-cutting).** For a non-existent or
  cross-workspace `project_id` the user has no membership for, `allow_permission`
  (`plane/app/permissions/base.py`) returns 403 before the view's `get_object_or_404`
  can return 404. Two tests assert 404 (`test_issue_app::...404_for_project_outside_workspace`,
  `test_project_bookmark::...handles_missing_project`). The executor correctly STOPPED
  rather than changing this — it affects every `@allow_permission`-decorated view, so
  it needs a deliberate, well-tested change.
- **010 — test-suite state isolation.** The FULL suite is non-deterministic: identical
  runs gave 2 vs ~12 failures because `transaction=True` tests (and `--reuse-db
--nomigrations`) leak committed rows across tests/runs — `test_projects.py` and
  `test_labels.py` fail in the full run but PASS in isolation. This must be fixed before
  "full suite green" is a reliable gate (for plan 005) or before pytest goes into CI
  (it would flake). Likely fix: ensure transactional tests truncate/clean up, or
  isolate them.
